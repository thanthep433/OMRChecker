# ADR 0004 — UI ต้นแบบของฟีเจอร์ "ตรวจข้อสอบ" (รีสกิน web_test/ เป็น Classwiser)

- **สถานะ:** ยอมรับ (Accepted)
- **วันที่:** 2569-07
- **เกี่ยวข้องกับ:** `web_test/` (Flask dev tool), ฟีเจอร์ `omr` ในอนาคต (Django)
- **อ้างอิง:** [`0003-omr-integration.md`](0003-omr-integration.md), design system ของ Classwiser

---

## 1. บริบท (Context)

`web_test/` เดิมเป็น Flask dev tool ธีมมืด 2 หน้า (อัปโหลด → ผลตรวจ) ไว้พิสูจน์ว่า OMR engine
อ่านกระดาษถูกและจูน `template.json` (เฟส 0–1 ของ `0003`). ต้องการปรับหน้าตา UI ให้ยึด design
system ของ Classwiser เพื่อเตรียมนำฟีเจอร์ "ตรวจข้อสอบ" ไปรวมกับ Classwiser จริง

คำถามคือ: จะรีสกินแค่ให้สวยขึ้น หรือจะให้เครื่องมือนี้ทำหน้าที่มากกว่านั้น?

## 2. การตัดสินใจ (Decision)

ทำให้ `web_test/` เป็น **"visual prototype" ที่เที่ยงตรงกับ Classwiser design system** โดย
HTML/CSS/โครงหน้าออกแบบให้ **ยกไปใช้ใน Django `omr` app (เฟส 3) ได้เลย** ไม่ใช่แค่รีสกินทิ้งเปล่า

- **IA:** ฟีเจอร์เป็น **เมนู top-nav ใหม่ "ตรวจข้อสอบ"** (แยกจาก "คะแนน") เพราะ flow หนัก —
  **Landing → wizard 6 สเต็ป** (วิชา·ชิ้นงาน·กระดาษ → เฉลย → อัปโหลด → ประมวลผล → ตรวจทาน → ยืนยัน)
- **ตรงไหนยังไม่มี data จริง (ต้องพึ่ง Classwiser DB) → เว้นว่าง/empty state อย่างตั้งใจ**
  ไม่ปั้น mock ปลอม: dropdown วิชา/ชิ้นงาน, คอลัมน์นักเรียนที่จับคู่, คลังเฉลย, การบันทึกคะแนนจริง
  เป็น placeholder ทั้งหมด แล้วไปเติม logic จริงใน Django
- **แกน engine ทำงานจริง:** อัปโหลด → อ่านกระดาษ → overlay + คำตอบ + คะแนน (คะแนนคำนวณจากกริดเฉลย
  ที่ครูกรอก ผ่าน `evaluation_json_override` ใน `runner.run_omr`)
- **หน้าตรวจทาน (หัวใจ):** ตารางสถานะรายแผ่นเป็นแกน + master-detail (กดแถวเปิด overlay ใหญ่ + แก้คำตอบ)
  ตาม pattern หน้า import รายชื่อของ Classwiser
- **ของ dev (เลือก template ต่างจากแก้ JSON):** การ *เลือกแบบกระดาษ* เป็น control ของผู้ใช้จริงในสเต็ป 1;
  ส่วนการ *แก้ `template.json` ดิบ* + engine log ซ่อนใต้ **toggle "โหมดพัฒนา" ตัวเดียวทั้งแอป (ปิด default)**

## 3. ผลที่ตามมา (Consequences)

**ข้อดี**
- ได้ทั้งเครื่องมือจูน template และต้นแบบ UI ที่ยก CSS/layout ไป Django ได้ — คุ้มกว่ารีสกินทิ้ง
- เห็น flow ตรวจทาน-ยืนยันที่เป็นหัวใจของฟีเจอร์ตั้งแต่ก่อนแตะ Django
- แยกชัดว่าอะไร "จริง" อะไร "รอต่อใน Django" (empty state) ลดการเข้าใจผิด

**ต้นทุน / สิ่งที่ต้องระวัง**
- มี UI ที่ยังไม่มี logic จริงรองรับ — ต้องกำกับด้วย empty state/ข้อความให้ชัดว่าเป็น placeholder
- state ของ wizard เก็บใน Flask session (dev tool) — ตอนย้าย Django ใช้โมเดล/ฟอร์มจริงแทน
- design token eyeball จาก screenshot — เมื่อมี stylesheet จริงของ Classwiser ให้แทนค่าให้ตรง

**ทางเลือกที่พิจารณาแล้วไม่เลือก**
- *รีสกิน 2 หน้าเดิมเฉยๆ* → ขาดหัวใจ (ตรวจทาน-ยืนยัน) ไม่คุ้มเป็นต้นแบบ
- *ปั้น mock data ให้ครบ* → ตัดทิ้ง เพราะทำให้แยกของจริง/ของปลอมยาก และเสี่ยงเข้าใจผิด
- *ทำเป็น tab ใต้ "คะแนน"* → flow หนักเกินกรอบ tab จึงแยกเป็นเมนู top-nav
